# Project Overview Document Design

## Goal

新增一份给项目 owner 自己看的内部总览文档，帮助在中断一段时间后快速恢复对 `SunCodexClaw` 当前架构、模块边界、配置模型和已做增强的整体理解。

目标不是替代现有 `README.md`，也不是写一份对外宣传稿，而是提供一份偏“系统地图”的文档。

## Audience

主要读者是当前项目 owner 自己。

默认假设：

- 读者知道这个项目是“飞书里的 Codex 机器人”
- 但不一定还记得当前有哪些本地增强已经落地
- 也不一定记得关键逻辑分别落在哪些文件里

## Scope

### In Scope

- 当前已经跑起来的 Feishu 实现
- 当前仓库真实存在的模块划分
- 当前多机器人运行模型和配置层次
- 当前已经落地的关键增强能力
- 已经明确讨论过、但尚未落地的后续方向

### Out of Scope

- 飞书后台逐步配置教程
- 日常运维命令清单
- 面向外部开发者的 marketing 介绍
- 企业微信版本的详细设计方案
- 与现有 README 重复的大段上手步骤

## Recommended Document Shape

文档建议保存为 `docs/project-overview.md`，采用 7 个固定部分：

1. `项目定位`
2. `运行架构`
3. `消息与任务链路`
4. `关键模块索引`
5. `配置与多机器人模型`
6. `当前已做增强`
7. `已规划能力与建议`

这 7 段的目的是：

- 先解释“这项目是什么”
- 再解释“它现在怎么工作”
- 然后解释“改动入口在哪里”
- 最后解释“它已经长成什么样，以及下一步可能往哪走”

## Section Intent

### 1. 项目定位

只回答三个问题：

- 这个项目解决什么问题
- 当前系统边界是什么
- 什么内容不应该写进这份文档

这部分应该短，不重复 README 的价值陈述。

### 2. 运行架构

用稳定的分层方式解释系统：

- 飞书接入层
- Codex 执行层
- 任务调度与上下文层
- 反馈层
- 本机运行与常驻层

每层都要同时给出：

- 责任
- 核心文件
- 与上下游的关系

### 3. 消息与任务链路

按顺序说明一条任务从进入到结束的路径，覆盖：

- 文本
- 图片
- 文件
- 语音
- 引用消息
- 最终回复和附件/文档回发

重点是让读者看到“消息驱动本机执行”的真实链路，而不是泛泛描述“机器人处理消息”。

### 4. 关键模块索引

按职责组织文件索引，而不是按目录平铺。每个模块都要回答：

- 它负责什么
- 首先应该去看哪个文件
- 哪些辅助文件和它关联

这部分应该帮助后续快速定位改动入口。

### 5. 配置与多机器人模型

解释：

- `config/feishu/*.json` 和 `config/secrets/local.yaml` 的分工
- 账号级配置如何叠加
- `default`、`second`、`yy_cooking_dev` 这种多机器人是如何区分的
- 哪些资源是隔离的，哪些是宿主级共享的

这部分需要诚实地写出“隔离已经做到哪里、还没做到哪里”。

### 6. 当前已做增强

系统化整理这段时间已经落地的真实增强，例如：

- 显式机器人名字与 `bot_open_id` 协调
- 群聊 mention carry
- 群文件放行
- 引用消息上下文
- 同群同发送者排队与 supersede 规则
- 按群名路由最终文本结果
- Markdown 卡片回复
- 监控面板

重点不是写变更日志，而是告诉未来的自己“项目现在比 README 原始印象多了哪些能力”。

### 7. 已规划能力与建议

只收录已经明确讨论过的后续方向，例如：

- 监控面板继续增强
- 多机器人隔离继续加固
- 权限继续精简
- 企业微信迁移可行性

这一段要明显区分“已实现”和“已讨论”。

## Source of Truth

正文内容应优先依据以下信息源整理：

- `README.md`
- `tools/feishu_ws_bot.js`
- `tools/lib/*.js`
- `docs/plans/*` 中已经存在的设计与实现计划
- 当前本机已经实际跑通的机器人形态

如果某能力只在讨论中出现、尚未落地，必须在文档里明确标注为“规划中”或“未实现”。

## Writing Principles

- 面向自己未来阅读，优先准确和可定位
- 不写大段营销语言
- 避免和 `README.md` 的安装、配置步骤重复
- 尽量用稳定术语，避免今天这样叫、明天换个名字
- 对风险点和边界保持诚实，不把“部分隔离”写成“完全隔离”

## Verification

完成文档后需要人工检查：

- 结构是否与批准的 7 段一致
- 是否能从文档快速跳到关键源码文件
- 是否和当前实现一致，没有把“计划中”写成“已完成”
- 是否明显减少了“回来后需要重新通读 README 和主脚本”的成本
